# Engineering Notes & Bug History

This document serves as the historical technical log for CaseLens AI. It tracks significant architectural hurdles, critical bugs, reproduction steps, stack traces, affected files, and their implemented solutions.

---

## 1. Vercel Build Crash: Top-Level OpenAI Instantiation

**Issue:**
The application successfully compiled locally but crashed consistently on Vercel during the `Collecting page data` optimization step. 

**Reproduction Steps:**
1. Push code to `main` to trigger a Vercel build.
2. Vercel runs `npm run build` which invokes `next build`.
3. During static page generation, Next.js evaluates all API routes.
4. The build process fatals with `Missing credentials`.

**Stack Trace:**
```text
Collecting page data using 1 worker ...
WARNING: GROQ_API_KEY is not set in the environment variables.
Error: Missing credentials. Please pass an `apiKey`, `workloadIdentity`, `adminAPIKey`, or set the `OPENAI_API_KEY` or `OPENAI_ADMIN_KEY` environment variable.
at module evaluation (.next/server/chunks/[root-of-the-server]__1usezk-._.js:1:1279)
> Build error occurred
Error: Failed to collect page data for /api/generate-overall-analysis
```

**Affected Files:**
- `lib/ai.ts`
- `app/api/generate-overall-analysis/route.ts`

**Implementation Notes (Solution):**
The `OpenAI` (Groq) client was being instantiated at the top level of the module (`const ai = new OpenAI({...})`). Because Vercel's build environment did not have the `GROQ_API_KEY` environment variable explicitly exposed to the static builder, the SDK threw a fatal missing credentials error. Refactored the codebase to strictly enforce **lazy instantiation**. The client is now instantiated dynamically inside a getter function (`getAiClient()`) or inside the `POST` handler, ensuring initialization only at runtime when a real HTTP request is processed.

## 2. Legacy ID Mapping Failures (The "q-idx" Bug)

**Issue:**
Older Multiple Choice Questions (MCQs) generated by early AI models lacked explicit `id` UUID fields. When the user submitted a practice attempt for these older cases, the Results page failed to map the user's attempt payload back to the original question data, causing the UI to display blank reviews or crash.

**Affected Files:**
- `lib/store.ts`
- `app/case/practice/page.tsx`

**Implementation Notes (Solution):**
The Zustand store's `submitPracticeAttempt` action relied strictly on `question.id`. Upgraded the backend and the Zustand store to implement a backward-compatible fallback ID generator. When finding questions, the codebase now maps against `q.id || 'q-' + idx`. This ensures legacy questions seamlessly map to the index-based IDs created on the practice page.

## 3. The "Ghost Session" Analytics Bug

**Issue:**
When a user completed a quiz and navigated rapidly from the Practice Page to the Results Page, the Results page would display "0 answered" despite the `localStorage` properly saving the attempts.

**Affected Files:**
- `lib/store.ts` (specifically `getLatestSession`)
- `app/case/results/page.tsx`

**Implementation Notes (Solution):**
The React component lifecycle unmounted abruptly during navigation, inadvertently triggering the auto-generation of an empty "Ghost Session" in the Zustand store. The Results page relied on a flawed `getLatestSession` utility that used heuristic date-sorting. Because the "Ghost Session" was created milliseconds *after* the actual completed session, the algorithm grabbed the empty session and rendered it. Deprecated purely date-based heuristic lookups. Upgraded the Next.js router to explicitly pass the exact `sessionId` query parameter directly in the URL (`/case/results?id=...&sessionId=...`).

## 4. TypeScript Strict Compilation Crashes

**Issue:**
Next.js Turbopack silently failed to compile, hiding updates from the UI and blocking deployments.

**Stack Trace:**
```text
Runtime ReferenceError
useMemo is not defined
    at IndividualResultsPageContent (app/case/results/page.tsx:55:21)
    
Runtime ReferenceError
getLatestSession is not defined
    at IndividualResultsPageContent.useMemo[latestSession] (app/case/results/page.tsx:67:5)
```

**Affected Files:**
- `app/case/results/page.tsx`
- `lib/store.ts`

**Implementation Notes (Solution):**
Duplicate helper functions, union-type mismatches (`McqQuestion | LegacyQuestion`), missing React imports (`useMemo`), and unimported interfaces (`QuestionReviewAnalysis`) violated strict type bindings. Executed a deep audit using `npx tsc --noEmit` across the codebase to expose and eradicate hidden strict-typing errors, enforce explicit imports, and ensure all interface properties aligned safely.

## 5. Monolithic Storage Bottlenecks

**Issue:**
Deeply nested array mutation algorithms were causing significant UI jank (stutter) and threatening imminent browser `QuotaExceededError` crashes (5MB limit).

**Affected Files:**
- `lib/store.ts`

**Implementation Notes (Solution Plan):**
Zustand was storing the entire global application state inside a single `localStorage` JSON blob. Every time a user clicked "Submit Answer", the application was forced to `O(N)` map over the entire history of the application just to serialize and save one object update. The architecture will transition away from monolithic JSON blobs towards asynchronous, relational `O(1)` Object Stores using IndexedDB.
